fix(hosting): close six hosting findings (B11/B14/B15/B17/B20/B21)

- **B14** — User-defined `headers()` from `next.config.js` (or Nuxt
  `routeRules`) now wins over the L3's hardcoded security headers when
  the user sets one of the typed slots (`x-frame-options`, HSTS,
  `x-xss-protection`, `referrer-policy`, `x-content-type-options`).
  Previously the user's `x-frame-options: DENY` was silently overridden
  by the L3 default `SAMEORIGIN`. Detection is case-insensitive on
  header names. Untouched typed slots stay at the L3 defaults.

- **B15** — Static-only deploys with a real `404.html` (Next.js
  `output: 'export'`, Astro/Hugo/Eleventy SSG) now serve the right 404
  body with status 404 instead of getting rewritten to `index.html`
  with status 200. The SPA fallback (404→200/index.html) is preserved
  for deploys that don't emit a `404.html`. Driven by a new
  `manifest.errorPages` field; the SPA adapter auto-populates it from
  filesystem detection.

- **B17** — `next.config.js` `assetPrefix` is now consumed end-to-end.
  Adapter reads the prefix from `.next/required-server-files.json`,
  stores it as `manifest.assetPrefix`, and the L3 emits matching
  CloudFront behaviors for `/<prefix>/_next/static/*`,
  `/<prefix>/_next/image*`, `/<prefix>/_next/data/*`, and
  `/<prefix>/_next/*` with a CF Function that strips the prefix before
  the build-id S3 lookup. Multi-zone Next.js apps (and any
  CDN-fronting setup with a path prefix) now work without manual
  CloudFront patches.

- **B20** — `CustomHeadersPolicy*` resources now use stable
  content-derived construct IDs (SHA-256 of the header fingerprint,
  truncated to 8 hex). Previously a positional counter caused every
  redeploy that reordered `manifest.headers[]` to create new policies
  - delete old ones, churning the account-wide CloudFront
    ResponseHeadersPolicy quota and risking leaks under failed
    rollbacks. With stable IDs, redeploys with unchanged content are a
    no-op and quota usage is bounded by unique header-set count.

- **B21** — Compute (SSR) cache behaviors now use a custom
  `CachePolicy` that honors origin `Cache-Control` instead of the
  AWS-managed `CACHING_DISABLED` policy. Previously every SSR request
  hit Lambda regardless of `s-maxage` / `stale-while-revalidate`
  emitted by the framework — silently breaking ISR/SWR for all five
  supported frameworks (Next, Nuxt, Astro, Remix, SvelteKit). The new
  policy:
  - keys on `Accept-Encoding` (compression negotiation) plus the four
    Next.js App Router headers (`rsc`, `next-router-prefetch`,
    `next-router-state-tree`, `next-router-segment-prefetch`) so RSC
    payloads don't pollute HTML cache entries
  - excludes cookies from the cache key (auth-bearing routes must opt
    out via `Cache-Control: private`)
  - clamps origin TTL between 0s and 1 year
  - defaults to no-cache when origin emits no Cache-Control (preserves
    safe behavior for SSR routes that forget the header)
  - enables CloudFront on-the-fly Brotli + Gzip

  POST/PUT/DELETE remain uncacheable per HTTP spec; CloudFront
  enforces this regardless of CachePolicy.

- **B11** confirmed wired (no code change needed): per-route Lambda@Edge
  associations are correctly attached on edge-route cache behaviors;
  Next.js `middleware.ts` runs inline in the OpenNext SSR Lambda
  (matches OpenNext's standard architecture). Verified live: 8 of 14
  cache behaviors on the Edge stress distribution have
  `LambdaFunctionAssociations.Quantity = 1` — exactly the 8 edge
  routes.

Tests: 355 passing (was 342 before this PR's commits and 347 before
this fix). Added 4 cases for the security-headers override (B14), 1
case for `errorPages` SPA fallback (B15), 6 cases for the
`assetPrefix` strip function + 1 for the prefixed-behavior emission
(B17), 1 case for the SSR CachePolicy properties (B21). No regressions.
