fix(hosting): close nine hosting findings (B11/B14/B15/B17/B20/B21/B22/B23/B24)

- B14 - User-defined `headers()` from `next.config.js` (or Nuxt
  `routeRules`) now wins over the L3's hardcoded security headers when
  the user sets one of the typed slots (`x-frame-options`, HSTS,
  `x-xss-protection`, `referrer-policy`, `x-content-type-options`).
  Previously the user's `x-frame-options: DENY` was silently overridden
  by the L3 default `SAMEORIGIN`. The user value is now parsed and
  injected directly into CloudFront's typed `securityHeadersBehavior`
  slots with `Override: true`. HSTS parses `max-age`,
  `includeSubDomains`, `preload`. Frame-options accepts `DENY` or
  `SAMEORIGIN`. Referrer-policy accepts the eight standard policy
  names. Detection is case-insensitive on header names. Untouched
  typed slots stay at the L3 defaults.

- B15 - Static-only deploys with a real `404.html` (Next.js
  `output: 'export'`, Astro/Hugo SSG) now serve the right 404 body
  with status 404 instead of getting rewritten to `index.html` with
  status 200. The SPA fallback (404 -> 200/index.html) is preserved
  for deploys that don't emit a `404.html`. Driven by a new
  `manifest.errorPages` field; the SPA adapter auto-populates it from
  filesystem detection.

- B17 - `next.config.js` `assetPrefix` is now consumed end-to-end.
  Adapter reads the prefix from `.next/required-server-files.json`,
  stores it as `manifest.assetPrefix`, and the L3 emits matching
  CloudFront behaviors for `/<prefix>/_next/static/*`,
  `/<prefix>/_next/image*`, `/<prefix>/_next/data/*`, and
  `/<prefix>/_next/*` with a CF Function that strips the prefix before
  the build-id S3 lookup. Multi-zone Next.js apps now work without
  manual CloudFront patches.

- B20 - `CustomHeadersPolicy*` resources now use stable
  content-derived construct IDs (SHA-256 of the header fingerprint,
  truncated to 8 hex). Previously a positional counter caused every
  redeploy that reordered `manifest.headers[]` to create new policies
  and delete the old ones, churning the account-wide CloudFront
  ResponseHeadersPolicy quota and risking leaks under failed rollbacks.

- B21 - Compute (SSR) cache behaviors now use a custom `CachePolicy`
  that honors origin `Cache-Control` instead of the AWS-managed
  `CACHING_DISABLED` policy. Previously every SSR request hit Lambda
  regardless of `s-maxage` or `stale-while-revalidate` emitted by the
  framework, silently breaking ISR/SWR for all five supported
  frameworks. The new policy keys on the four Next.js App Router
  headers (`rsc`, `next-router-prefetch`, `next-router-state-tree`,
  `next-router-segment-prefetch`), excludes cookies (auth-bearing
  routes opt out via `Cache-Control: private`), clamps origin TTL
  between 0s and 1 year, and enables CloudFront on-the-fly Brotli +
  Gzip. POST/PUT/DELETE remain uncacheable per HTTP spec.

- B11 - confirmed wired (no code change needed). Per-route
  Lambda@Edge associations are correctly attached on edge-route cache
  behaviors. Next.js `middleware.ts` runs inline in the OpenNext SSR
  Lambda. Verified live: edge-runtime routes on the App Router stress
  have `LambdaFunctionAssociations.Quantity = 1`.

- B22 - Synthesized cache behaviors for `manifest.headers[]` patterns
  now inherit the catch-all's origin choice (compute when
  `hasCompute`, static otherwise). Previously, header rules whose
  source pattern had no existing behavior unconditionally synthesized
  a static-origin behavior, which 403'd on SSR routes because the S3
  bucket has no file at e.g. `/headers-test`. CloudFront resolves
  behaviors first-match-wins, so the 403'ing static behavior shadowed
  the catch-all's compute origin.

- B23 - `customHeadersBehavior.customHeaders[]` no longer contains
  security-named headers. CloudFront rejects deploys (HTTP 400) when
  reserved security headers (`x-frame-options`, HSTS,
  `x-xss-protection`, `referrer-policy`, `x-content-type-options`,
  CSP) appear in the custom-headers array. They must instead be set
  via the typed `securityHeadersBehavior`. User values are still
  applied via the typed slots (see B14); they're now filtered out of
  the custom-headers list to satisfy CloudFront's validation.

- B24 - The B21 SSR `CachePolicy` no longer enables Brotli/Gzip while
  also whitelisting `Accept-Encoding` in `headerBehavior`. CloudFront
  rejects that combination (HTTP 400 - "EnableAcceptEncodingGzip
  cannot be set to true when accept-encoding header is whitelisted").
  CloudFront handles content-negotiation cache-keying implicitly when
  the encoding flags are enabled.

Tests: 319 passing in the hosting package. Added 5 cases for the
security-headers override (B14/B23), 1 case for `errorPages` SPA
fallback (B15), 6 cases for the `assetPrefix` strip function + 1 for
the prefixed-behavior emission (B17), 1 case for the SSR CachePolicy
properties (B21/B24).

Live verification: deployed three test apps (Nuxt/Nitro, Next App
Router, Next Pages Router) to a clean AWS account. All B14/B17/B21
probes pass on App Router and Nuxt. Pages Router has a known
limitation when `i18n` is configured: the locale prefix
`/:nextInternalLocale(...)` causes the adapter's source-pattern
classifier to skip the rule. Fix tracked separately as B25.
